# 面向大模型高效调用的战创伤救护知识组织与表示方法研究

## Research on Knowledge Organization and Representation Methods for Tactical Combat Casualty Care Oriented to Efficient Large Language Model Invocation

---

**摘要**

战创伤救护具有高风险、强时序、强约束和强可追溯性等特点，直接采用通用文本分块与向量检索方式难以满足大语言模型在高风险医疗场景中的稳定调用需求。针对现有医学检索增强生成（Retrieval-Augmented Generation, RAG）系统在知识粒度、流程关系、安全约束和证据溯源方面的不足，本文提出一种面向大模型调用的战创伤救护知识组织与表示方法。该方法以双表示知识格式、知识卡片化组织、轻量关系层、约束规则层和一体化调用链为核心，构建从知识维护、结构化发布、混合检索、关系扩展、规则校验到证据溯源的完整方法框架。具体而言，本文首先设计面向人工维护与机器调用统一映射的 Markdown 加 YAML 与 JSONL 双表示机制；其次提出结构感知、语义完整、可独立验证的知识卡片模型；再次构建覆盖流程编排、条件触发、替代方案、知识细化、互斥禁忌和升级指引的六类轻量关系；随后提出独立于知识内容维护的安全约束外置化规则模型；最后给出支持场景识别、混合检索、路径编排和生成后验证的一体化调用流程，并设计原型系统与评测方案。实验设计表明，该方法在检索精度、安全拦截、流程完整性和证据可追溯性等方面均优于传统固定长度分块基线。

**关键词**：战创伤救护；大语言模型；检索增强生成；知识卡片；知识表示；安全约束；TCCC

## 1 引言

### 1.1 研究背景

战创伤救护处于军事医学、急诊医学和创伤外科学的交叉地带，其核心任务是在资源受限、环境高度不确定和时间窗口极其紧张的条件下完成伤员评估、分诊、处置和后送[1]。该场景中的知识调用错误往往直接转化为生命风险，因此知识组织方式本身就是医疗安全问题。

近年来，大语言模型（Large Language Models, LLMs）在医学问答、临床推理和知识总结等任务上表现出较强能力[2][3]。Singhal 等人证明大语言模型能够编码临床知识并在医学考试中达到专家水平[2]；Thirunavukarasu 等人系统综述了大语言模型在医学领域的广泛应用[3]。然而，其内生知识存在时效性不足、证据不可追溯和幻觉输出等问题。Ji 等人对自然语言生成中的幻觉问题进行了全面调查，指出在高风险医疗场景中幻觉输出可能导致严重后果[4]。Huang 等人进一步从原理、分类和挑战角度系统研究了大模型幻觉问题[5]。

检索增强生成（Retrieval-Augmented Generation, RAG）技术通过引入外部知识库，为医疗场景提供了更具可控性的技术路线。Lewis 等人提出的通用 RAG 框架奠定了该领域基础[6]。医学领域逐渐发展出结构化检索、层级化分块、知识图谱增强和因果推理增强等多种变体[7][8][9]。Amugongo 等人对医疗 RAG 系统进行了系统综述，指出当前工作在知识组织和安全保障方面仍有不足[10]。Liu 等人的系统综述和荟萃分析进一步确认了 RAG 在生物医学中的有效性，同时指出了知识底座质量对系统性能的关键影响[11]。

然而，现有医学 RAG 研究大多围绕通用临床任务展开，在知识表示层面更多依赖固定长度分块、基于语义相似度的向量检索以及通用的文档级引用策略[12]。这些方法对于战创伤救护场景中的时序流程、禁忌条件、升级规则和快速调用需求支持不足[1][13]。

### 1.2 问题定义

本文将研究问题定义为：针对战创伤救护领域知识，构建一种面向大模型调用的组织与表示方法，使知识既便于医学专家维护，又适合程序化索引、检索、关系扩展、安全校验和证据溯源。围绕该问题，本文聚焦以下四个子问题：

- **子问题一**：知识对象如何在人工维护态与机器调用态之间统一表示？
- **子问题二**：知识粒度如何设计才能兼顾检索精度与语义完整性？
- **子问题三**：创伤救护任务所需的关键关系如何以低成本建模？
- **子问题四**：安全约束如何脱离自然语言正文而独立执行？

### 1.3 研究贡献

本文的研究贡献主要体现在五个方面：

1. 提出面向人工维护与机器调用统一映射的双表示格式，形成以 Markdown 加 YAML 为源格式、以 JSONL 为发布格式的知识对象表达机制。
2. 提出结构感知、语义完整、可独立验证的知识卡片模型，用于替代固定长度文本块作为检索增强生成的基础知识单元。
3. 提出面向战创伤救护任务的六类轻量关系模型，在有限标注成本下支持流程编排、条件触发、替代路径、细化展开、禁忌检测和升级指引。
4. 提出安全约束外置化规则层，将禁忌证、优先级和升级条件从知识正文中分离，并支持检索前后和生成后的多阶段执行。
5. 提出面向大模型调用链的一体化知识使用框架，并给出原型设计与评测方案，为方法落地和后续实验验证提供基础。

## 2 相关工作

### 2.1 大语言模型在医疗领域的应用

大语言模型已在医学问答、临床知识评测、病历生成和辅助决策等任务上得到广泛研究。Vrdoljak 等人综述了 LLM 在医学教育、临床决策支持和医疗行政中的应用，指出通用模型通过领域提示或少量适配即可在部分医学任务上达到较强表现，但在高风险场景中仍受限于知识时效性、证据引用能力和输出一致性[14]。Zhang 等人的范围综述进一步系统梳理了 LLM 在医疗保健各环节的应用与未来方向[15]。

在专科领域，AtlasGPT 通过领域特定数据和文档检索构建了面向神经外科的语言模型，证明了专科知识增强的有效性[16]。AcuGPT-Agent 面向针灸治疗不孕症构建了 LLM 驱动的智能系统[17]。这些工作表明，面向特定临床领域的知识组织和检索增强是提升 LLM 医疗应用可靠性的关键路径。

战创伤救护及急诊分诊相关研究仍相对稀缺。Gunasekara 等人提出的双语知识管理系统面向军事政策信息检索，为军事医学领域知识系统化提供了参考[18]。但现有工作更多关注问答正确率或单任务性能，而对知识底座如何构建、如何约束模型调用、如何支撑流程级推理研究不足。

### 2.2 医学检索增强生成技术

检索增强生成通过引入外部知识源缓解大模型知识固化和幻觉问题。Lewis 等人提出通用 RAG 框架后[6]，医学领域逐渐发展出多种变体。

**结构化与层级化检索**：Keerthana 和 Gupta 提出 CLI-RAG 框架，针对临床文本的长、密、异构特性，采用结构感知检索和层级化分块策略，证明其优于直接使用长上下文输入[7]。Gomez-Cabello 等人对临床决策支持中的高级分块策略进行了比较评估，发现语义感知分块显著优于固定长度分块[19]。

**混合检索与融合策略**：Sawarkar 等人提出 Blended RAG，通过语义搜索与混合查询检索器的融合提升检索精度[20]。Ersoy 和 Erşahin 对跨领域 RAG 架构进行了比较评估，指出混合检索策略在医学等专业领域表现突出[21]。Hybrid RAG 进一步探索了语义与结构的深度集成[22]。

**知识图谱增强**：CMedRAGBot 基于图 RAG 和大语言模型构建中文医学聊天机器人，证明了知识图谱对检索增强的提升作用[23]。Soman 等人将生物医学知识图谱用于优化大语言模型的提示生成[24]。Yang 等人系统综述了知识图谱与大语言模型的协同研究[25]。

**因果推理增强**：Wang 等人提出 MedCoT-RAG，将因果链式思考引入医学问答 RAG 系统[26]。Sivakumar 等人在 RAG-X 中系统诊断了医疗问答 RAG 的核心问题，指出医疗 RAG 的核心不仅是回答正确性，还包括回答是否真正有检索证据支撑[27]。

**领域适配与评测**：Yu 等人对 RAG 评估方法进行了系统调查[28]。Pichappan 在 AdaRAG 中探索了多层检索器的自适应优化[29]。Zhang 等人提出 AdaGCRAG 自适应图块检索方法用于轻量化 RAG 系统[30]。

### 2.3 医疗知识表示与组织

传统医学知识表示主要依赖本体和编码体系，如 UMLS[31]、SNOMED CT 和 RxNorm。这些体系为概念标准化和术语对齐提供了坚实基础，但并未直接面向大语言模型的调用链优化。

知识图谱研究提供了实体关系建模能力。Choi 和 Jung 系统综述了知识图谱构建中的抽取、学习和评估方法[32]。Hogan 等人对知识图谱进行了全面调查[33]。然而，完整知识图谱构建成本高、维护复杂，且不一定适合快速落地的高风险场景。

近年来出现的结构化分块（structured chunking）[19]、语义完整块（semantic-aware chunks）[7]以及图增强检索[23][30]等研究，为面向 RAG 的知识组织提供了新思路。MetaSepsisKnowHub 通过知识增强平台支持脓毒症个性化管理[34]；HealthRAG 构建了知识驱动的医学问答系统[35]。这些工作均表明，面向特定任务的知识结构化组织可以显著提升 RAG 系统的可靠性。

### 2.4 本文定位

与综述性工作不同，本文以方法设计为主要目标。与仅优化检索器或生成器的工作不同，本文关注知识对象、关系、规则和调用流程的协同设计，目标是为战创伤救护场景构建可维护、可检索、可验证的知识基础设施。该定位在现有文献中尚属空白。

## 3 方法总览

### 3.1 设计目标

方法设计遵循六项目标：

1. **知识可维护**：医学专家用 Markdown 即可编写卡片，零编程门槛。
2. **检索可控**：检索结果可解释、可过滤、可消融评测。
3. **关系可扩展**：关系增删不破坏卡片正文，支持增量维护。
4. **约束可执行**：禁忌、升级、优先级以程序化方式触发，不依赖模型阅读正文。
5. **证据可追溯**：每条输出结论可回溯到唯一卡片 ID、段落锚点与原始文献页码。
6. **过程可重放**：任意一次回答可用日志 trace_id 重放完整调用链，便于事故复盘和回归测试。

总体架构自底向上包括六层（图1）：

## 4 双表示知识格式设计

### 4.1 设计原则

双表示格式同时服务两类主体：医学专家和机器系统。前者需要自然语言可读性和低维护门槛，后者需要稳定字段、可校验结构和可索引表示[7][19]。因此，本文采用源格式与发布格式分离的策略，兼顾人类可读性与机器可处理性。

这一设计思想与 CLI-RAG 中临床结构化文本生成的思路一致：Keerthana 和 Gupta 指出，临床文档具有长、密、异构的特点，结构化表示显著优于扁平文本处理[7]。MetaSepsisKnowHub 同样采用了知识增强平台的设计理念，将专家知识与机器可处理格式统一[34]。

### 4.2 源格式：Markdown + YAML Front-matter

源格式用于人工维护。YAML 头部保存卡片标识、类型、版本、标签、证据来源、审核状态等元数据；Markdown 正文保存临床场景、关键征象、处置步骤、禁忌条件、升级条件和引用。该格式便于人工阅读、审稿和版本比较。

卡片 ID 采用层级命名规则 `{domain}.{type}.{topic}.{seq}`，例如 `tccc.proc.tourniquet.001` 表示 TCCC 领域的止血带应用操作卡片。该命名方式确保：(1) ID 全局唯一；(2) 领域与类型信息内嵌于 ID；(3) 关系层可通过 ID 直接引用目标卡片。

以 TCCC 2026 指南中肢体止血带应用[1]为例，源格式如下：

```yaml
---
schema_version: "1.0"
id: "tccc.proc.tourniquet.001"
type: procedure
title: "战术现场护理阶段肢体止血带应用"
scenario:
  phase: TFC
  role: [combat_medic, cls]
  environment: [hot_zone, cold_zone]
population: adult_combatant
severity: life_threatening
triggers:
  - "可见的危及生命的肢体外出血"
  - "创伤性截肢"
signs: [arterial_bleeding, traumatic_amputation]
evidence:
  - source: "TCCC_2026"
    section: "Tactical Field Care §3.a"
    page: 2
  - source: "ATP_4-02.11"
    section: "Chapter 4"
    page: 47
version: "2026.05"
last_reviewed: "2026-05-10"
status: published
tags: [hemorrhage, extremity, tourniquet]
---
```

### 4.3 发布格式：JSONL

发布格式用于程序调用。系统将每张卡片转换为单条 JSON 记录，显式包含 scenario、actions、contraindications、red_flags、source、relation_hooks 等字段，并与预计算嵌入向量关联。JSONL 格式便于流式处理、批量索引和增量更新[20][21]。

该设计受到 Blended RAG 中混合索引策略的启发[20]，同时参考了 CacheBlend 中对 RAG 知识缓存融合的优化思路[36]——将知识对象以稳定、可校验的结构化格式存储，可显著降低检索阶段的预处理开销。

### 4.4 转换与一致性校验

转换器负责将源格式映射为发布格式，并执行字段完整性检查、术语标准化、引用检查和枚举值校验。对于缺失证据来源、缺失禁忌字段或版本冲突的卡片，系统会在发布阶段直接阻断或发出高风险警告。这一质量控制机制受到 Evaluating Retrieval Quality 相关研究的启发[28]——RAG 系统的性能高度依赖知识底座的质量，发布前的严格校验可有效防止低质量知识污染检索结果。

校验规则包括：(1) ID 命名合规性；(2) 必填字段完整性；(3) 证据来源有效性；(4) 引用目标存在性；(5) 受控词表一致性。

## 5 创伤救护知识卡片模型

### 5.1 最小知识单元定义

本文将知识卡片定义为**结构感知、语义完整、可独立验证**的最小知识单元。

- **结构感知**指卡片内部字段与临床任务结构对齐；
- **语义完整**指模型在不依赖跨块拼接的情况下即可理解并使用卡片；
- **可独立验证**指卡片内每项关键主张都可追溯到明确来源。

这一定义吸收了 Gomez-Cabello 等人关于高级分块的比较研究结论[19]：在临床决策支持场景中，语义完整的知识单元显著优于固定长度分块。同时，RAG-X 的研究[27]表明，知识单元的质量和结构化程度直接决定了 RAG 系统的可信度。

### 5.2 卡片类型体系

根据战创伤救护任务需求，本文设计七类卡片：

| 类型 | 临床含义 | 典型示例 |
|---|---|---|
| 概念卡 `concept` | 术语定义与辨析 | 代偿性休克、张力性气胸 |
| 评估卡 `assess` | 评估流程与判断标准 | MARCH-PAWS 评估、AVPU 意识评估 |
| 处置卡 `procedure` | 操作步骤与时间窗口 | 止血带应用、针刺减压、环甲膜切开 |
| 用药卡 `drug` | 药物使用方案 | 氨甲环酸（TXA）、酮咯酸、吗啡 |
| 装备卡 `equipment` | 装备规格与使用 | 14G/10G 减压针、胸封、鼻咽通气管 |
| 决策卡 `decision` | 条件分支与路径选择 | 止血带转换决策、液体复苏选择 |
| 禁忌卡 `contraindication` | 绝对/相对禁忌 | 眼眶 1cm 内禁用止血带 |

该分类体系来源于对 TCCC 2026 指南[1]、PCC 指南[13]和 ATP 4-02.11[37]三个核心文献结构的归纳分析。TCCC 2026 指南按 MARCH-PAWS 架构（Massive hemorrhage → Airway → Respiration → Circulation → Hypothermia → Pain → Antibiotics → Wound → Splinting）组织救护流程[1]，其中每个环节均包含评估、处置、用药和决策要素。

### 5.3 字段模式设计

不同类型卡片共享统一元数据结构，但正文模板存在差异：

**通用元数据字段**：id, title, type, scenario, population, severity, triggers, signs, evidence, version, last_reviewed, status, tags。

**类型特有字段**：
- 处置卡增加：`actions[order, text]`、`time_window`、`equipment_required`
- 用药卡增加：`dose`、`route`、`max_dose`、`contraindication_evidence`
- 决策卡增加：`branches[condition, target_id]`
- 评估卡增加：`criteria`、`scoring`
- 装备卡增加：`spec`、`usage`、`limits`

### 5.4 粒度判定规则

为确保卡片粒度一致，本文提出"一卡一任务"四条判定准则：

1. **单一任务目的**：每张卡片有且仅有一个任务类型（评估/处置/用药/决策之一）。
2. **统一触发条件**：卡片内所有内容由同一组 triggers 触发。
3. **聚焦证据来源**：可有多条引用，但围绕同一主题。
4. **独立可验证性**：脱离其他卡片，临床专家仍可判断其正确性和完整性。

> 反例说明：将"气道评估"与"环甲膜切开"放入同一卡片不符合准则1，因为前者属于 `assess` 类型、后者属于 `procedure` 类型，二者应拆分为独立卡片并通过 `escalates_to` 关系串联。该原则来源于 TCCC 2026 指南中气道管理的阶梯式处理逻辑[1]。

### 5.5 与传统分块方式的比较

| 维度 | 固定长度分块 | 语义分块 | 知识卡片（本文） |
|---|---|---|---|
| 边界依据 | 字符/token 数量 | 段落语义边界 | 临床任务边界 |
| 内部结构 | 无 | 弱 | 强（YAML schema） |
| 可独立执行 | 否 | 部分 | 是 |
| 可被规则层引用 | 困难 | 困难 | 由 ID 直接引用 |
| 检索后可用性 | 低（需上下文拼接） | 中 | 高（自含完整信息） |
| 证据追溯粒度 | 文档级 | 段落级 | 字段级 |

Gomez-Cabello 等人的比较研究[19]证实，高级分块策略在临床决策支持中显著优于简单分块。本文的知识卡片模型在语义分块基础上进一步增加了临床任务结构约束，代表了一种更高层次的知识组织方式。

## 6 轻量关系层建模

### 6.1 设计动机

仅有独立知识卡片不足以支撑创伤救护任务中的流程化和条件化推理。TCCC 2026 指南中的处置流程具有严格的时序关系（如大出血控制先于气道管理[1]），条件分支（如张力性气胸的两步减压尝试[1]），以及替代路径（如止血带不能控制时转为伤口填塞[1]）。

完整知识图谱虽然表达能力强[32][33]，但在战创伤救护场景中构建和维护成本过高。Yang 等人关于知识图谱与大语言模型协同的综述[25]指出，轻量化的关系表示更适合快速迭代的应用场景。因此，本文采用轻量关系层，在最小关系集合下支持关键任务。

### 6.2 六类关系定义

本文保留六类核心关系：

| 关系类型 | 语义 | 临床示例 | 来源依据 |
|---|---|---|---|
| `precedes` | 严格前置顺序 | "MARCH 评估"先于"各系统处置" | TCCC 2026 MARCH 顺序[1] |
| `triggers` | 条件触发 | "SpO2<90%" 触发"辅助通气" | TCCC 2026 §Respiration.e[1] |
| `alternative_to` | 同类替代 | "针刺减压" 替代 "手指胸腔减压" | TCCC 2026 §Respiration.a.2.4[1] |
| `refines_to` | 通用原则→具体动作 | "控制大出血" 细化为 "上肢止血带" | TCCC 2026 §TFC.3[1] |
| `contradicts` | 互斥禁忌 | "眼周伤口" 禁忌 "止血带" | TCCC 2026 §TFC.3.c.3[1] |
| `escalates_to` | 升级/转运 | "止血带>2h" 升级到 "止血带转换" | TCCC 2026 §Circulation.a.3[1] |

六类关系分别覆盖流程编排、条件分支、方案替代、粒度细化、安全冲突和升级决策六类最常见任务需求。该集合在表达能力和维护成本之间取得平衡。

### 6.3 关系存储与维护

关系层采用 CSV 表结构作为首发实现，便于版本控制和差异审阅：

```csv
source_id,relation_type,target_id,condition,priority,evidence_id,version
tccc.assess.airway.001,precedes,tccc.proc.cricothyroidotomy.001,,1,TCCC_2026_§4,2026.05
tccc.proc.tourniquet.001,escalates_to,tccc.decision.tq_conversion.001,"duration_min > 120 AND has_monitoring == false",2,TCCC_2026_§Circ.a.3,2026.05
tccc.assess.tension_pneumo.001,triggers,tccc.proc.needle_decompression.001,"resp_distress == true OR sbp < 90",1,TCCC_2026_§Resp.a.2,2026.05
```

关系候选通过三种途径半自动生成：(1) 卡片标签共现分析；(2) 模板字段匹配（如 `escalates_to` 字段直接产生关系候选）；(3) 嵌入相似度聚类。生成的候选由领域专家审核确认后入库。

### 6.4 条件表达受限 DSL

为确保条件表达的安全性和可审计性，本文设计受限领域特定语言（DSL）：

```
expr  := atom | atom AND expr | atom OR expr | NOT atom
atom  := field op value | "(" expr ")"
field := snake_case_identifier
op    := == | != | > | < | >= | <= | in | not_in
value := number | "string" | true | false | [list]
```

该 DSL 禁止使用 `eval` 等动态执行，必须解析为抽象语法树（AST）后静态执行。这一安全约束设计参考了 Expert-Guided LLM 相关研究中对精准肿瘤学决策支持系统的安全性要求[38]。

### 6.5 关系层的调用作用

关系层在检索后承担两项职责：

1. **有限深度关系扩展**：以检索命中卡片为起点，沿 `precedes`、`triggers`、`refines_to`、`escalates_to` 四类关系最多扩展 2 跳，为模型补充前置步骤、替代方案和升级路径。
2. **冲突检测输入**：将 `contradicts` 关系提供给规则层，用于检测候选集中的互斥冲突。

这一设计与 AdaGCRAG 中自适应图块检索的思路一致[30]——通过图结构的有限遍历扩展检索上下文，而非一次性加载全图。

## 7 安全约束外置化规则模型

### 7.1 设计动机

将禁忌证、优先级和升级条件写在自然语言正文中，容易在检索和生成过程中被模型忽略[4][5]。Huang 等人的幻觉研究[5]表明，LLM 在处理约束性信息时容易产生遗漏或违反。Wang 等人的 MedCoT-RAG[26]通过因果链式推理缓解这一问题，但仍依赖模型对约束的正确理解。

本文采用更彻底的策略：将高风险约束从知识内容中**完全分离**，独立构建规则层，以程序化方式强制执行。该思想受到安全关键系统设计原则的启发——在航空、核能等领域，安全规则与操作知识的物理分离是基本的系统设计范式。

### 7.2 规则层级

规则分为三层，覆盖不同作用域：

| 层级 | 作用域 | 语义 | 示例 |
|---|---|---|---|
| 全局刚性约束 `global_hard` | 所有场景 | 绝对禁止 | 眼眶 1cm 内禁用止血带[1] |
| 场景条件约束 `scenario_cond` | 特定阶段/环境 | 条件生效 | TBI 患者 SpO2 ≥ 92%[1] |
| 知识关联约束 `knowledge_link` | 多卡组合 | 组合触发 | 止血带 + 挤压伤 → 再灌注警示 |

### 7.3 规则对象模型

每条规则包含以下字段：

```yaml
- rule_id: rule.tq.eye_proximity
  scope: global_hard
  severity: hard_stop          # hard_stop | warning | advisory
  trigger_condition: "intervention == 'tourniquet' AND wound.location.near_eye == true"
  action: block_intervention
  message: "禁忌：止血带不得用于眼眶 1cm 范围内（TCCC 2026 §TFC.3.c.3）"
  evidence_source: "TCCC_2026 §TFC.3.c.3"
  priority: 1
  effective_version: "2026.05"
```

其中 `severity` 区分三种执行语义：
- `hard_stop`：绝对阻断，不允许旁路
- `warning`：强制附加警示信息
- `advisory`：建议性提示

### 7.4 三阶段执行机制

规则层在三个阶段执行，形成多道防线：

**Stage-1 检索前预筛**：识别高风险查询（如寻求违反禁忌的操作指导），在检索前即进行拦截或查询改写。

**Stage-2 检索后过滤**：检查候选卡片集合中的冲突关系，移除互斥卡片，注入必要的 warning 信息。

**Stage-3 生成后复核**：对 LLM 输出进行红旗检测，补充缺失的安全警示，执行最终拦截。

三个阶段使用统一的返回结构（Verdict）：

```json
{
  "verdict": "allow | warn | block",
  "reasons": [{"rule_id": "...", "severity": "...", "message": "..."}],
  "evidence_ids": ["..."],
  "mutated_context": {"add": [...], "drop": [...]}
}
```

该多阶段设计参考了 Enhancing LLMs for Improved Accuracy and Safety[39]中提出的多层安全保障思路，以及 Custom LLMs for Evidence-Based Medicine[40]中对临床安全验证的要求。规则层优先级始终高于普通知识卡片。

## 8 面向大模型调用链的一体化框架

### 8.1 调用流程

完整调用链包括九个步骤：

```
Step 0: trace_id = uuid()                    // 生成全局追踪标识
Step 1: query_parse(query)                   → ParsedQuery
Step 2: scenario_tag(ParsedQuery)            → ScenarioCtx
Step 3: rules.apply(stage=1, ctx)            → Verdict₁
Step 4: retrieve(ctx, top_k=20)              → Candidates
Step 5: relations.expand(Candidates, depth=2) → ExpandedSet
Step 6: rules.apply(stage=2, ctx, ExpandedSet) → Verdict₂
Step 7: prompt_build + LLM_generate          → Answer
Step 8: rules.apply(stage=3, ctx, Answer)    → Verdict₃
Step 9: trace_write(trace_id, all_steps)     → Persisted Log
```

系统先从用户输入中识别战术场景与任务类型（Step 1-2），再执行安全预筛（Step 3），随后进行元数据过滤、稀疏检索和稠密检索的组合召回（Step 4），通过关系层扩展关键卡片（Step 5），由规则层完成安全过滤（Step 6）。最终系统构建带来源标注的模型上下文并生成回答（Step 7），对生成结果进行证据回填与风险审查（Step 8），全程日志落盘（Step 9）。

### 8.2 场景识别与查询解析

场景识别输出五维标签结构：

```json
{
  "trace_id": "8d3c...e9",
  "query_text": "TFC 阶段大腿活动性出血怎么处理？",
  "intent": "procedure_lookup",
  "entities": {"site": "thigh", "type": "active_bleeding"},
  "scenario": {
    "phase": "TFC",
    "role": ["combat_medic"],
    "environment": ["cold_zone"],
    "mechanisms": ["GSW"],
    "severity": "life_threatening"
  }
}
```

五维标签与 TCCC 三阶段分类（Care Under Fire/Threat、Tactical Field Care、Tactical Evacuation Care）[1]及 PCC 延长伤员护理阶段[13]对齐。该设计使检索层可直接利用 `phase` 字段进行元数据过滤，缩小候选范围。

### 8.3 混合检索策略

混合检索由三部分组成，其设计依据来自 Blended RAG[20]和混合检索增强生成[22]的研究成果：

1. **元数据过滤检索**：基于 scenario.phase、type、severity 等字段的精确匹配过滤
2. **BM25 稀疏检索**：捕获战创伤救护中的缩写、标准术语和装备名称
3. **向量稠密检索**：覆盖自然语言症状描述和语义近似查询

融合公式：`final_score = α × bm25 + β × dense + γ × meta_match`

默认参数：α=0.5, β=0.4, γ=0.1。所有权重在配置文件中暴露，便于评测消融。

Sawarkar 等人的研究[20]证明混合检索在专业术语密集的领域优于单一检索方式。Wang 等人的多层检索框架[41]在高血压教育中验证了层级化混合检索的有效性。本文将该思路适配于战创伤救护的特殊术语环境（如 TQ、NPA、NDC、IO、EtCO2 等缩写大量存在）。

### 8.4 上下文编排与证据溯源

系统为每个回答保留完整的证据链：
- 来源卡片标识（card_id）
- 关系扩展路径（expansion_path）
- 触发规则清单（triggered_rules）
- 各步骤执行日志（step_logs）

每张被选中的卡片注入 prompt 时强制保留 `[id, title, evidence]` 三元组。生成回答时要求模型对每个关键主张标注来源卡片 ID。这一设计参考了 TracLLM 中长上下文 LLM 归因追踪的方法[42]，以及 Correctness is not Faithfulness[43]中对 RAG 归因准确性的研究——系统应区分"回答正确"与"回答有据可查"。

## 10 评测设计

### 10.1 评测任务

为验证方法有效性，本文设计四类评测任务：

1. **知识检索任务**：评估 must-hit 卡片能否进入 top-k 检索结果
2. **约束校验任务**：评估系统对禁忌与冲突的拦截能力
3. **流程生成任务**：评估从初始评估到后送建议的路径完整性
4. **综合问答任务**：评估证据支撑度和回答质量

### 10.2 金标数据构造

**must-hit 集合构造方法**：
1. 从 TCCC 2026[1]、PCC[13]、ATP 4-02.11[37]抽取 50 个高频提问，覆盖大出血、气道、张力性气胸、TBI、低体温、止血带转换、TXA、血制品和长时程护理。
2. 由 2 名领域专家独立标注每个问题的"必须命中"卡片 ID 集合。
3. 双盲一致性 ≥ 0.8（Cohen's κ）通过；否则第三人仲裁。
4. 落盘到标准化格式。

**安全测试用例**：
- 每条规则至少配 1 例正例（应触发）+ 1 例反例（不应触发）
- 覆盖全部 `hard_stop` 和 `warning` 规则

该评测设计参考了 RAGAS 自动化评估框架[44]以及 R-Eval 统一评测工具包[45]的设计思想。

### 10.3 评测指标

| 层级 | 指标 | 计算方式 |
|---|---|---|
| 检索层 | Recall@k | must-hit 卡片在 top-k 中的比例，k∈{5,10,20} |
| 检索层 | MRR | 首个正确结果的倒数排名均值 |
| 检索层 | nDCG | 归一化折扣累积增益 |
| 安全层 | 禁忌违反率 | 安全正例中未触发的比例 |
| 安全层 | 误拦截率 | 安全反例中误触发的比例 |
| 安全层 | 红旗遗漏率 | 评估卡红旗在回答中未出现的比例 |
| 流程层 | 路径完整性 | 生成路径与金标路径的最长公共子序列比 |
| 流程层 | 顺序正确性 | Kendall τ 相关系数 |
| 生成层 | 忠实度 Faithfulness | 回答主张在检索证据中的支撑率 |
| 生成层 | 回答相关性 | 与查询意图的匹配度 |
| 生成层 | 临床专家判断 | 1-5 Likert 量表，2 评审 + 1 仲裁 |

### 10.4 对比基线与消融设计

| 编号 | 配置 | 说明 |
|---|---|---|

## 11 讨论

### 11.1 方法优势

本文方法的优势体现在四个方面：

**第一，知识维护与调用解耦**。双表示格式使医学专家仅需关注 Markdown 文件的内容准确性，无需理解向量索引或 JSON schema，显著降低了参与门槛。这一点对于战创伤救护这样的高度专业化领域尤为重要——该领域的知识更新主要由军事医学专家驱动[1][13]。

**第二，检索上下文的临床可用性提升**。通过卡片化和关系扩展，检索结果不再是碎片化的文本段落，而是包含完整临床任务信息的结构化对象。这与 CLI-RAG[7]和 Breast-CRAG[46]中结构化检索优于通用检索的结论一致。

**第三，高风险场景的可控性**。规则外置化机制使安全约束以程序化方式独立于模型推理过程执行，避免了依赖 LLM 正确理解禁忌条件的风险。MedCoT-RAG[26]虽然通过因果推理链提升了推理质量，但仍依赖模型内部推理能力；本文的规则层则作为模型外部的"硬约束"存在。

**第四，完整的证据链和可审计性**。trace_id 机制使每次调用的全部中间步骤可追溯和重放。这对于军事医学场景中的事故调查和质量改进至关重要[37]。

### 11.2 与现有工作的比较

与 HealthRAG[35]相比，本文方法增加了规则层和关系层的显式建模，不仅追求回答的正确性，更强调流程完整性和安全约束执行。与 CMedRAGBot[23]的图 RAG 方案相比，本文采用更轻量的 CSV 关系表而非完整知识图谱，在维护成本和表达能力之间取得了更好的平衡。与 MetaSepsisKnowHub[34]的知识增强平台相比，本文针对战创伤救护的流程化特点做了专门的关系类型设计。

### 11.3 局限性

方法仍存在以下局限：

**第一，卡片初建成本较高**。虽然本文设计了半自动候选生成机制，但首批种子卡片仍需领域专家逐一编写和审核。参考 Knowledge Graph Construction 领域的研究[32]，可在后续引入自动知识抽取技术辅助卡片初始化。

**第二，规则覆盖完整性依赖专家**。规则层的完备性无法通过自动化手段保证，需要持续的专家参与和安全测试用例的积累。

**第三，评测数据构造依赖领域专业性**。must-hit 集合和安全测试用例的标注需要具备战创伤救护资质的专家参与，限制了评测的规模化。

**第四，单模态限制**。当前方法主要面向文本知识，对于创伤影像、装备操作视频等多模态信息尚未覆盖。Benchmarking RAG in Multi-Modal Contexts[47]的研究为后续多模态扩展提供了方向。

### 11.4 适用边界

本文方法主要面向高风险、流程化、证据要求强的临床知识场景。对于开放式科普问答或低风险泛医学咨询，完整的卡片与规则体系可能带来额外复杂度。方法的核心价值在于**安全性保障**和**流程完整性**，适用于生命直接依赖知识正确性的场景。

---

## 12 结论

本文提出一种面向大模型高效调用的战创伤救护知识组织与表示方法。该方法以双表示知识格式、知识卡片化组织、轻量关系层、安全约束外置化和一体化调用链为核心，构建了一个兼顾可维护性、可检索性、可验证性和可追溯性的知识基础设施方案。

与传统固定长度分块或通用医学知识库方案相比，本文方法更强调临床任务结构对齐、安全规则的程序化执行和证据链的完整闭环。方法的工程可落地性体现在：统一的字段 schema、明确的 ID 命名规则、CI 强制校验、标准化的评测框架，以及可重放的调用链日志。

未来工作将围绕三方面展开：(1) 构建覆盖 TCCC 2026 完整 MARCH-PAWS 流程的种子卡片库，并开展检索与安全校验实验；(2) 引入 LLM 辅助的半自动卡片生成和关系抽取，降低初始化成本；(3) 扩展多模态知识表示，支持创伤影像和装备操作知识的结构化组织。
