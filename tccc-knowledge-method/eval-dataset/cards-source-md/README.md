# 卡片源格式 (MD+YAML) — 专家审核入口

> 这是论文 §4.2 设计的**源格式**目录。
> 此处的 `.md` 文件是给医学专家/医学审核员阅读和修改的版本,
> 不是给机器索引用的(机器用的是 `cards-dist/cards.jsonl`)。

## 1. 这里有什么

44 张知识卡片的 MD+YAML 形态,**与 JSON 一一对应、双向无损**:

| 子目录 | 卡片来源 | 数量 |
|---|---|---|
| `HEM/` | TCCC 2026 大出血模块 | 24 |
| `PCC/` | Prolonged Casualty Care CPG ID:91 | 11 |
| `ATP/` | ATP 4-02.11 March 2026 | 9 |

## 2. 每张卡的结构

```
---                              ← YAML front-matter 开始
card_id: PCC-SEPSIS-001            机器关心的元数据
title: ...                         (专家不必动这里)
module: PCC
related_cards: [ABX-002]
source_refs:
  - source_id: JTS-PCC-CPG-91
    section: ANTIBIOTICS - PCC
    locator: p.40 Table 14
    quote: "Minimum - Moxifloxacin..."
---                              ← YAML 结束

# 卡片标题
                                 ← 以下都是专家可以直接读和改的 Markdown

## 1. 核心问题
## 2. 输出目标
## 3. 适用场景与边界            ← 含适用场景/前置条件/不覆盖范围
## 4. 触发与识别                ← 含触发条件/识别要点/临床征象
## 5. 关键动作 / 流程           ← Markdown 表格
## 6. 决策点                    ← 如果...则...的列表
## 7. 复评 / 终点 / 禁忌 / 错误 ← 4 个聚合块
## 8. 证据来源(展开)           ← 包含原文引用 quote
## 9. 评分要点(LLM 评测)
## 10. 备注
```

## 3. 专家工作流

```
┌──────────────────────────────────────────────────────────────┐
│ 1. 工程方提 PR: 把 cards-source-md/* 推到 GitHub             │
└──────────────────────────────────────────────────────────────┘
                           ↓
┌──────────────────────────────────────────────────────────────┐
│ 2. 专家在 GitHub Web 直接打开 .md 文件,GitHub 会渲染成漂亮  │
│    的页面 (像看 Wiki),不需要任何编辑器或 JSON 知识          │
└──────────────────────────────────────────────────────────────┘
                           ↓
┌──────────────────────────────────────────────────────────────┐
│ 3. 专家审核: 4 种行为                                        │
│   (a) 直接 GitHub 评论 / suggestion (轻量改 1-2 行)          │
│   (b) Web 编辑器改文字 (改临床内容)                          │
│   (c) 在 PR 评论提整张卡的修改建议                           │
│   (d) 把 review_status 从 draft 改成 expert_reviewed         │
└──────────────────────────────────────────────────────────────┘
                           ↓
┌──────────────────────────────────────────────────────────────┐
│ 4. 工程方接受 PR,跑 `make from-md` 把 .md 转回 .json        │
└──────────────────────────────────────────────────────────────┘
                           ↓
┌──────────────────────────────────────────────────────────────┐
│ 5. CI 闸口: `make ci` 自动跑 validate / build / index        │
│    任一项失败 → 阻断合并,工程修后再来                       │
└──────────────────────────────────────────────────────────────┘
```

## 4. 双向转换(对工程方)

```bash
# JSON 改了 → 重新生成 MD 给专家看
make to-md

# 专家改了 MD → 转回 JSON 入库
make from-md
make ci          # 闸口校验,通过才能合并
```

`from_md.py` 转换是 **100% 无损**的(实测 44/44 一致),专家在 .md 改了什么,
.json 就更新什么,YAML front-matter 中的字段也会同步。

## 5. 专家不需要懂的事

- ❌ JSON 语法(花括号、引号、逗号)
- ❌ Schema 字段必填性(机器闸口会检查)
- ❌ 卡片 ID 命名规则(已经定好,不要动 YAML 头里的 `card_id`)

## 6. 专家需要确认的事

| 字段 | 怎么改 |
|---|---|
| 第 4 节"触发与识别" | 是不是与指南一致?有没有漏关键征象? |
| 第 5 节"关键动作" | 步骤顺序、剂量、时间窗对不对?表格直接改单元格 |
| 第 6 节"决策点" | 条件是否准确?分支是否完整? |
| 第 7 节"禁忌 / 警示" | **这一项最关键,漏一条就是事故** |
| 第 8 节"证据来源" | 章节号、页码是否引对? |
| YAML `review_status:` | 审核完后从 `draft` 改成 `expert_reviewed` |

## 7. 实际效果对比

同一张 PCC-SEPSIS-001 卡片:

- **JSON 形态**:38 行,全是花括号引号(给机器看)
- **MD+YAML 形态**:119 行,但其中 80 行是渲染好的临床速查表(给专家看)

专家审一张 MD 卡片大约 **5-10 分钟**;
让专家审一张 JSON 卡片至少 **30 分钟**而且大概率会改坏文件。
