# 面向大模型高效调用的战创伤救护知识组织与表示方法（改进方案）

> 文档版本：v1.0（基于用户提供的《战创伤救护知识组织方法.pdf》原稿改进）
> 评估对象：用户原稿（共 11 节、15 页）
> 参考语料：TCCC Guidelines 01 May 2026、Prolonged Casualty Care Guidelines (JTS CPG ID:91)、ATP 4-02.11 (March 2026)、CLI-RAG、RAG-X、MedCoT-RAG 等
> 适用阶段：方法定稿 + 原型工程化落地

---

## 0. 评价结论先行（对原稿《战创伤救护知识组织方法.pdf》的判断）

**总体判断：方向可行，框架成立，但作为"工程可落地的技术方案"仍偏论文化，需要补四件硬东西。**

| 维度 | 原稿表现 | 是否可行 | 主要问题 |
|---|---|---|---|
| 问题定义（4 个子问题） | 清晰、收敛 | ✅ 可行 | 与 TCCC/PCC/ATP 4-02.11 真实结构对得上 |
| 双表示格式（MD+YAML / JSONL） | 思路成立 | ✅ 可行 | 缺字段级 schema、缺 ID 命名规则、缺版本与审核状态机 |
| 知识卡片模型（7 类） | 分类合理 | ✅ 可行 | 缺各类型字段差异化定义、缺粒度判定规则、缺与 TCCC 章节的映射 |
| 六类轻量关系 | 集合精炼 | ✅ 基本可行 | 缺 `condition` 的形式化语法、缺与规则层的边界划分 |
| 安全约束外置化（三层规则） | 思想正确 | ⚠️ 需补强 | 三阶段执行点缺触发协议、缺 `hard_stop / warning / advisory` 的统一返回结构 |
| 调用链（7 步） | 流程完整 | ⚠️ 需补强 | 缺"场景识别"与"查询解析"的具体输出 schema，缺可观测性约定 |
| 评测设计 | 指标到位 | ✅ 可行 | 缺金标数据构造方法、缺 must-hit 集合的来源约定 |
| 可工程化程度 | 偏论文 | ❌ 不足 | 没有目录结构、没有 CI 校验流水线、没有最小可运行原型 |

**改进策略（本方案做的事）：**
1. 把每个抽象概念**全部落到 schema、目录结构、文件名、字段类型**。
2. 用 **TCCC 2026 真实条文**（大出血→气道→张力性气胸→循环→TBI→低体温）做样例填充，使方案可被同行检验。
3. 增加**安全约束的统一执行协议**（三阶段、统一返回值、可旁路日志）。
4. 增加**最小可运行原型骨架**（目录树 + 关键脚本接口签名），把"原型设计"从一段话变成可以 `git clone` 起步的工程。
5. 给出**评测金标的构造 SOP**，让评测从"指标列表"变成"可以做的事"。

## 2. 设计目标（5 条 + 1 条新增）

源自原稿 §3.1，新增第 6 条以呼应工程落地：

1. **可维护**：医学专家用 Markdown 即可写卡片，零编程门槛。
2. **可控检索**：检索结果可解释、可过滤、可消融。
3. **可扩展关系**：关系增删不破坏卡片正文。
4. **可执行约束**：禁忌、升级、优先级以代码方式触发，不依赖模型读懂正文。
5. **可追溯证据**：每条结论都能回到唯一卡片 ID + 段落锚点 + 原始文献页码。
6. **【新增】可重放**：任意一次回答都能用日志 `trace_id` 重放完整调用链，便于事故复盘与回归测试。

## 4. 双表示知识格式（细化到字段）

### 4.1 源格式（专家维护态）：Markdown + Front-matter YAML

文件路径：`source/tccc/proc/tourniquet/001.md`

```markdown
---
schema_version: 1.0
id: tccc.proc.tourniquet.001
type: procedure              # concept|assess|procedure|drug|equipment|decision|contraindication
title: 战术现场护理阶段肢体止血带应用
scenario:
  phase: TFC                 # CUF | TFC | TACEVAC | PCC
  role: [combat_medic, cls]
  environment: [hot_zone, cold_zone]
population: adult_combatant
severity: life_threatening
triggers:
  - "可见的危及生命的肢体外出血"
  - "创伤性截肢"
signs: [arterial_bleeding, traumatic_amputation]
evidence:
  - source: TCCC_2026
    section: "Tactical Field Care §3.a"
    page: 2
  - source: ATP_4-02.11
    section: "Chapter 4"
    page: 47
version: 2026.05
last_reviewed: 2026-05-10
reviewed_by: [reviewer_a, reviewer_b]
status: published            # draft|review|published|deprecated
tags: [hemorrhage, extremity, tourniquet]
---

## 适用场景
战术现场护理阶段（TFC），出现危及生命的肢体外出血或创伤性截肢。

## 关键征象
- 动脉性出血（喷射状/搏动性）
- 创伤性截肢
- 出血点解剖位置适合止血带

## 处置步骤
1. 选用 CoTCCC 推荐型止血带。
2. **直接置于皮肤上**，距出血点近端 5-7 cm（2-3 inches）。
3. 收紧至出血停止且远端搏动消失。
4. 若一根不能止血，**并排再上一根**（不是叠放）。
5. 标注上止血带时间，记录于 TCCC 伤员卡。

## 禁忌与边界
- 不在关节上方使用。
- 已超过 6 小时的止血带，**未具备监护与实验室条件不得解除**。

## 升级条件
- 出血未控制 → 升级到伤口填塞 + 加压包扎组合（见 tccc.proc.wound_packing.001）。
- 已上止血带 > 2 小时但未到上级救治：触发 `escalates_to: tccc.decision.tq_conversion.001`。
```

## 5. 知识卡片模型（细化）

### 5.1 7 类卡片与差异化字段

| 类型 `type` | 必填扩展字段 | 典型用途 | 样例 |
|---|---|---|---|
| `concept` | definition, synonyms | 定义类术语 | "代偿性休克" |
| `assess` | criteria, scoring | 评估流程 | MARCH-PAWS、AVPU |
| `procedure` | actions[order,text], time_window, equipment | 操作流程 | 止血带应用、针刺减压 |
| `drug` | dose, route, indications, contraindications, evidence | 用药 | 氨甲环酸、酮咯酸 |
| `equipment` | spec, usage, limits | 装备 | 14G/10G 减压针 |
| `decision` | branches[condition,target_id] | 分诊/决策 | 止血带转换决策 |
| `contraindication` | rule_ref | 禁忌（多映射到规则层） | 眼周止血禁用 |

### 5.2 卡片 ID 命名规则（新增）

`{domain}.{type}.{topic}.{seq}`

- `domain`：`tccc | pcc | atp40211 | local`
- `type`：见 §5.1
- `topic`：snake_case 主题词
- `seq`：3 位顺序号

例：`pcc.assess.march_paws.001`、`tccc.drug.txa.001`、`tccc.decision.tq_conversion.001`。

### 5.3 粒度判定规则（新增："一卡一任务"四条）

一张卡片**有且仅有一个**：
1. **任务目的**（评估 / 处置 / 用药 / 决策，其一）；
2. **触发条件集合**（同一组 triggers）；
3. **证据来源段落**（可多条引用，但围绕同一主题）；
4. **可独立验证**：脱离其他卡片，临床可执行或可判断。

> 反例：把"气道评估 + 环甲膜切开"放一卡是错的，前者是 `assess`、后者是 `procedure`，应拆为两卡，用 `escalates_to` 串联。

### 5.4 与传统分块对比（保留原稿观点，加一张表）

| 维度 | 固定长度 chunk | 语义 chunk | **知识卡片** |
|---|---|---|---|
| 边界依据 | 字符数 | 段落语义 | **临床任务** |
| 字段化 | 无 | 弱 | **强（YAML schema）** |
| 可独立执行 | 否 | 部分 | **是** |
| 可被规则层引用 | 难 | 难 | **由 ID 直接引用** |

## 6. 轻量关系层（细化 condition 语法）

### 6.1 六类关系（保留）

| 关系 | 语义 | 典型例 |
|---|---|---|
| `precedes` | 严格前置 | "评估气道" precedes "环甲膜切开" |
| `triggers` | 条件触发 | "SpO2<90%" triggers "辅助通气" |
| `alternative_to` | 同类替代 | "针刺减压" alternative_to "胸腔闭式引流" |
| `refines_to` | 通用→具体 | "控制大出血" refines_to "上止血带" |
| `contradicts` | 互斥禁忌 | "眼周伤口" contradicts "止血带应用" |
| `escalates_to` | 升级转运 | "止血带 >2h" escalates_to "止血带转换" |

### 6.2 存储结构（CSV，便于 git diff）

文件 `relations/relations.csv`：

```csv
source_id,relation_type,target_id,condition,priority,evidence_id,version
tccc.assess.airway.001,precedes,tccc.proc.cricothyroidotomy.001,,1,tccc.evidence.airway.001,2026.05
tccc.proc.tourniquet.001,escalates_to,tccc.decision.tq_conversion.001,"duration_min > 120 AND has_monitoring == false",2,tccc.evidence.tq.001,2026.05
tccc.proc.tourniquet.001,contradicts,tccc.contraind.eye_proximity.001,"location.near_eye == true",1,tccc.evidence.tq.002,2026.05
tccc.assess.tension_pneumo.001,triggers,tccc.proc.needle_decompression.001,"resp_distress == true OR sbp < 90",1,tccc.evidence.tp.001,2026.05
```

### 6.3 condition 受限 DSL（新增）

只允许下列形式，不允许任意表达式：

```
expr     := atom | atom AND expr | atom OR expr | NOT atom
atom     := field op value | "(" expr ")"
field    := snake_case            # 如 duration_min, sbp, has_monitoring
op       := == | != | > | < | >= | <= | in | not_in
value    := number | "string" | true | false | [list]
```

> 解析器必须给出 AST，禁止使用 `eval`。这是为了**安全审计**。

### 6.4 关系层与规则层的边界（修订）

- **关系层**回答"**下一步该读哪张卡**"——影响**检索召回与上下文拼装**。
- **规则层**回答"**这张卡此刻能不能用 / 要不要警告**"——影响**最终生成是否被允许**。

> 同一冲突可在两层都有体现（例：眼周伤口 + 止血带），但二者**职责不重叠**：关系层负责把卡片关联起来，规则层负责裁决是否阻断。

## 7. 安全约束外置化规则层（统一执行协议）

### 7.1 三层规则（保留）

| 层级 | 含义 | 例 |
|---|---|---|
| 全局刚性约束 `global_hard` | 任何场景禁止 | 眼眶 1cm 内禁用止血带 |
| 场景条件约束 `scenario_cond` | 特定阶段/环境生效 | TBI 患者 SpO2 目标 ≥92% |
| 知识关联约束 `knowledge_link` | 多卡组合触发 | 已上止血带 + 怀疑挤压伤 → 警示再灌注 |

### 7.2 规则对象模型（统一 schema）

文件 `rules/rules.yaml`：

```yaml
- rule_id: rule.tq.eye_proximity
  scope: global_hard
  severity: hard_stop
  trigger_condition: "intervention == 'tourniquet' AND wound.location.near_eye == true"
  action: block_intervention
  message: "禁忌：止血带不得用于眼眶 1cm 范围内（TCCC 2026）"
  evidence_source: "TCCC_2026 §TFC.3.c.3"
  priority: 1
  effective_version: "2026.05"

- rule_id: rule.tbi.spo2_target
  scope: scenario_cond
  severity: warning
  trigger_condition: "patient.tbi_severity in ['moderate','severe'] AND measure.spo2 < 92"
  action: prompt_supplemental_oxygen
  message: "中重度 TBI：维持 SpO2 ≥ 92%，建议补充供氧（TCCC 2026 Respiration §c-d）"
  evidence_source: "TCCC_2026 §TFC.5.c-d"
  priority: 2

- rule_id: rule.tq.duration_no_monitoring
  scope: knowledge_link
  severity: hard_stop
  trigger_condition: "tourniquet.duration_min > 360 AND has_monitoring == false"
  action: block_removal
  message: "止血带在位 >6 小时且无监护/实验室条件，禁止解除"
  evidence_source: "TCCC_2026 §Circulation.a.3"
  priority: 1
```

```
                  │                              │                              │
                  ▼                              ▼                              ▼
   apply_rules(query_ctx) ──► apply_rules(query_ctx, candidates) ──► apply_rules(query_ctx, answer)
                  │                              │                              │
                  └────────── 统一返回 ──────────┴──────────── 统一返回 ─────────┘

   Verdict := {
     verdict: "allow" | "warn" | "block",
     reasons: [ {rule_id, severity, message} ],
     evidence_ids: [...],
     mutated_context: { add: [...], drop: [...] }   // 可选，仅 Stage-2 使用
   }
```

### 7.4 优先级仲裁

并发触发时按 `priority` 升序裁决，`hard_stop` 永远高于 `warning`、`warning` 高于 `advisory`。

## 8. 一体化调用链（7 步 + 全程 trace_id）

```
0. trace_id = uuid()
1. query_parse(query)            → ParsedQuery
2. scenario_tag(ParsedQuery)     → ScenarioCtx{phase, role, environment, mechanisms, severity}
3. rules.apply(stage=1, ctx)     → Verdict₁  // 高风险查询拒答 / 改写
4. retrieve(ctx)                 → Candidates  // BM25 + 向量 + 元数据过滤
5. relations.expand(Candidates)  → ExpandedSet  // precedes / triggers / refines_to ≤ 2 跳
6. rules.apply(stage=2, ctx, ExpandedSet) → Verdict₂  // 过滤冲突卡 / 注入 warning
7. prompt_build → LLM → answer
8. rules.apply(stage=3, ctx, answer) → Verdict₃  // 红旗补全 / 阻断不合规答案
9. write trace: runs/<trace_id>/{step_1.json ... step_9.json}
```

### 8.1 ParsedQuery / ScenarioCtx schema（新增）

```json
{
  "trace_id": "8d3c...e9",
  "query_text": "TFC 阶段大腿活动性出血怎么处理？",
  "intent": "procedure_lookup",
  "entities": {"site": "thigh", "type": "active_bleeding"},
  "scenario": {
    "phase": "TFC",
    "role": ["combat_medic"],
    "environment": ["cold_zone"],
    "mechanisms": ["GSW"],
    "severity": "life_threatening"
  }
}
```

### 8.2 混合检索（保留并细化权重）

`final_score = 0.5 * bm25 + 0.4 * dense + 0.1 * meta_match`，所有权重在 `config/retrieval.yaml` 暴露给评测消融。

### 8.3 上下文编排（带证据锚点）

每张被选中的卡片注入 prompt 时**强制保留** `[id, title, evidence]` 三元组，用于回答里的引用回填。

## 11. 评测设计（可执行 SOP）

### 11.1 四类任务（保留原稿）+ 金标构造 SOP（新增）

**must-hit 集合构造步骤：**
1. 从 TCCC 2026 / PCC / ATP 4-02.11 抽取 50 个高频提问（覆盖大出血、气道、张力性气胸、TBI、低体温、止血带转换、TXA、血制品、长时程护理）。
2. 由 2 名领域审稿人独立标注每个问题的"必须命中"卡片 ID 集合（来自本仓 `dist/cards.jsonl`）。
3. 双盲一致性 ≥ 0.8（Cohen's κ）通过；否则进入第三人仲裁。
4. 落盘到 `eval/goldset/must_hit.csv`，列：`qid, query, must_hit_ids, scenario_phase, severity`。

**safety_cases 构造：**
- 每条规则至少配 1 例**正例**（应触发）+ 1 例**反例**（不应触发），落盘到 `eval/goldset/safety_cases.yaml`。
- 例：止血带 + 眼周伤口（应 hard_stop）；止血带 + 大腿活动性出血（应 allow）。

### 11.2 指标（保留并明确口径）

| 层 | 指标 | 计算口径 |
|---|---|---|
| 检索 | Recall@k / MRR / nDCG / Exclusive Hit Rate | k ∈ {5,10,20} |
| 安全 | Contraindication Violation Rate | safety_cases 正例中未触发的比例 |
| 安全 | False Block Rate | safety_cases 反例中误触发的比例 |
| 安全 | Missing Red Flag Rate | 评估卡红旗未在回答中出现的比例 |
| 流程 | Pathway Completeness / Order Correctness | 与金标流程的最长公共子序列 |
| 生成 | Faithfulness / Answer Relevance / Clinician Judgment | 1-5 Likert，2 评审 + 1 仲裁 |

### 11.3 消融（保留）

- B0：固定长度 chunk + 向量
- B1：结构化 chunk + 混合检索
- B2：本方法去掉规则层
- B3：本方法去掉关系层
- **M：完整本方法**

每组报告 §11.2 全部指标，验证卡片 / 关系 / 规则各自增益。

## 12. 实施流程（Step-by-Step 操作步骤）

> 给具体的人（医学专家 + 工程）能照着做的清单，而不是"建议构建"。

### Phase 1：方法定稿（1 周）
1. 评审本文档，冻结 schema v1.0、关系六类、规则三层。
2. 落盘 `config/vocab.yaml` v1。
3. 选定 3 篇基准来源：TCCC 2026、PCC CPG ID:91、ATP 4-02.11。

### Phase 2：种子卡片（2 周）
4. 由 2 名领域专家分别从 TCCC 2026 抽取 30 张种子卡（覆盖 MARCH 全链路）。
5. 转换器跑通 `make validate build`，CI 全绿。
6. 写 10 张 PCC 卡（评估 + 长时程护理流程）。

### Phase 3：关系与规则（1 周）
7. 基于 30+10 张卡片，专家 + 半自动（标签共现 + 模板匹配）生成关系候选 ≥ 80 条，人工确认入 `relations.csv`。
8. 写 15 条核心规则（涵盖 TQ 禁忌、TBI SpO2、TQ 转换 2h/6h、张力性气胸再次减压等）。

### Phase 4：调用链原型（1 周）
9. 实现 `retrieve / relate / rules / orchestrate`（混合检索用 BM25 + 任意 sentence-encoder）。
10. 跑通 1 条端到端 query：`runs/<trace_id>/` 完整落盘。

### Phase 5：评测金标 + 实验（2 周）
11. 按 §11.1 构造 50 个 must-hit + 30 个 safety_cases。
12. 跑 §11.3 四组消融，输出 `eval/reports/`。
13. 写实验小节，回填进论文。

### Phase 6：扩展（开放）
14. 多模态：影像/装备图像卡（type=`equipment_visual`），用 `evidence.image_path` 关联。
15. 把 PCC 75 页全部卡片化（持续增量）。

## 13. 风险与边界（修订原稿 §10）

| 风险 | 缓解 |
|---|---|
| 卡片初建成本高 | 半自动抽取 + 模板（专家只做审定）；优先 30 张种子覆盖 MARCH 主流程 |
| 规则覆盖不全 | 用 CI 强制：`safety_cases` 中每条规则必须有正反例 |
| 受控词表演化 | `vocab.yaml` 走 PR + 评审，破坏性变更升 schema 大版本 |
| 多语言（中英混排） | 卡片正文允许中文；字段值（phase / role / 词表）一律英文常量，避免歧义 |
| 多模态扩展 | 在 `equipment` 类卡上加 `media:[{path,modality,caption}]`，不破坏现有 schema |
